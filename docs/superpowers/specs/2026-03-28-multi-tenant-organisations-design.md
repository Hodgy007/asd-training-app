# Multi-Tenant Organisations ‚Äî Design Spec

**Date:** 2026-03-28
**Project:** asd-training-app
**Status:** Approved

---

## Overview

Add multi-tenant organisation support to the ASD training app. The charity operates as the top-level authority (SUPER_ADMIN). It onboards partner organisations, each managed by an Org Admin. Users belong to organisations and see only their own org's data, announcements, and assigned training modules.

---

## 1. Data Model

### New model: `Organisation`

```prisma
model Organisation {
  id               String   @id @default(cuid())
  name             String
  slug             String   @unique
  active           Boolean  @default(true)
  allowedModuleIds String[] // Postgres text array ‚Äî org-level module defaults
  allowedRoles     String[] // Roles org admin may assign; validated at app layer against Role enum
  logoUrl          String?
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt

  users         User[]
  announcements Announcement[]
}
```

Note: `allowedRoles` is stored as `String[]` (Postgres text array) rather than a join table for simplicity. Application-layer validation against the `Role` enum is required on every write to prevent invalid values.

### New model: `Announcement`

```prisma
model Announcement {
  id             String    @id @default(cuid())
  title          String
  body           String    // Markdown
  active         Boolean   @default(true)
  organisationId String?   // null = global (SUPER_ADMIN only)
  createdById    String
  expiresAt      DateTime?
  createdAt      DateTime  @default(now())
  updatedAt      DateTime  @updatedAt

  organisation Organisation? @relation(fields: [organisationId], references: [id], onDelete: SetNull)
  createdBy    User          @relation("AnnouncementsCreated", fields: [createdById], references: [id])
}
```

### Changes to `User`

```prisma
// New fields added to existing User model
organisationId     String?   // null for SUPER_ADMIN only
allowedModuleIds   String[]  // empty = inherit org defaults
mustChangePassword Boolean   @default(false)

organisation   Organisation? @relation(fields: [organisationId], references: [id])
announcements  Announcement[] @relation("AnnouncementsCreated")  // back-relation required by Prisma
```

**Password field convention:** The existing `password` field remains non-nullable (consistent with current behaviour). Users created with a temporary password store the bcrypt hash of that password. SSO-only users store an empty string `""` (matching existing convention documented in CLAUDE.md).

### Role enum

Remove `ADMIN`. Add `SUPER_ADMIN`, `ORG_ADMIN`, `STUDENT`, `INTERN`, `EMPLOYEE`.

```prisma
enum Role {
  SUPER_ADMIN
  ORG_ADMIN
  CAREGIVER
  CAREER_DEV_OFFICER
  STUDENT
  INTERN
  EMPLOYEE
}
```

### Migration path

**Order matters ‚Äî must be done in sequence:**

1. Run a raw SQL update to migrate existing data before touching the enum:
   ```sql
   UPDATE "User" SET role = 'SUPER_ADMIN' WHERE role = 'ADMIN';
   ```
2. Create a "Legacy" organisation via a Prisma seed script (not raw SQL) so the `id` is generated by `cuid()`, consistent with the schema:
   ```ts
   await prisma.organisation.create({
     data: {
       name: 'Legacy',
       slug: 'legacy',
       active: true,
       allowedModuleIds: ['module-1','module-2','module-3','module-4','module-5'],
       allowedRoles: ['CAREGIVER','CAREER_DEV_OFFICER'],
     },
   })
   ```
3. Assign all existing `CAREGIVER` / `CAREER_DEV_OFFICER` users to the Legacy org:
   ```sql
   UPDATE "User" SET "organisationId" = (SELECT id FROM "Organisation" WHERE slug = 'legacy')
   WHERE role IN ('CAREGIVER', 'CAREER_DEV_OFFICER');
   ```
4. Apply schema changes via `npx prisma db push` (safe once no `ADMIN` rows remain).

**Do not run `prisma db push` before step 1** ‚Äî the destructive enum change will fail or prompt a database reset if `ADMIN` rows exist.

## 2. Role Hierarchy & RBAC

```
SUPER_ADMIN
    ‚îî‚îÄ‚îÄ ORG_ADMIN (scoped to one org)
            ‚îî‚îÄ‚îÄ CAREGIVER | CAREER_DEV_OFFICER | STUDENT | INTERN | EMPLOYEE
```

### Capability matrix

| Capability | SUPER_ADMIN | ORG_ADMIN | Leaf roles |
|---|---|---|---|
| Create / manage organisations | ‚úì | ‚Äî | ‚Äî |
| Set org allowed roles & modules | ‚úì | ‚Äî | ‚Äî |
| Create org admins + set first-time password | ‚úì | ‚Äî | ‚Äî |
| Create users within own org | ‚Äî | ‚úì | ‚Äî |
| Assign roles (from org's allowed list) | ‚Äî | ‚úì | ‚Äî |
| Assign modules per user | ‚Äî | ‚úì | ‚Äî |
| Post global announcements | ‚úì | ‚Äî | ‚Äî |
| Post org-scoped announcements | ‚Äî | ‚úì | ‚Äî |
| View cross-org training stats | ‚úì | ‚Äî | ‚Äî |
| View own org training stats | ‚Äî | ‚úì | ‚Äî |
| ASD / Careers training | ‚Äî | ‚Äî | ‚úì (per assignment) |
| Children / observations / reports | ‚Äî | ‚Äî | CAREGIVER only |

### Post-login redirect by role

Each role lands on their home on login:

| Role | Redirect target |
|---|---|
| `SUPER_ADMIN` | `/super-admin` |
| `ORG_ADMIN` | `/admin` |
| All leaf roles | `/dashboard` |

### RBAC helpers (`lib/rbac.ts`)

New helpers:

- `isSuperAdmin(session)` ‚Äî role === SUPER_ADMIN
- `isOrgAdmin(session)` ‚Äî role === ORG_ADMIN
- `isLeafRole(session)` ‚Äî any of the five leaf roles
- `getEffectiveModules(user, org)` ‚Äî returns `user.allowedModuleIds` if non-empty, otherwise `org.allowedModuleIds`; falls back to all ASD training module IDs if both empty. Note: career-focused orgs must have careers module IDs in their `org.allowedModuleIds` ‚Äî the SUPER_ADMIN is responsible for configuring this when creating the org. The fallback (ASD modules only) is intentionally conservative and only applies when org defaults are not yet configured.

**Existing helpers to update:**

- `isAdmin(session)` ‚Üí now checks `SUPER_ADMIN` (replaces ADMIN check)
- `canAccessCareers(session)` ‚Üí `CAREER_DEV_OFFICER` only (ADMIN removed; SUPER_ADMIN does not access training routes)
- `canAccessCaregiving(session)` ‚Üí `CAREGIVER` only (ADMIN removed; SUPER_ADMIN does not access training routes)

All existing call sites that relied on `ADMIN` passing `canAccessCareers` / `canAccessCaregiving` must be audited ‚Äî none should remain after the role migration.

**New leaf roles (`STUDENT`, `INTERN`, `EMPLOYEE`) and module access gating:**

These roles do not map to a fixed training area ‚Äî their access is purely determined by `getEffectiveModules`. Training pages must be updated to replace role-based `canAccessCareers` / `canAccessCaregiving` checks with a module-presence check: a user can access `/training/*` if their effective modules include any ASD module ID, and `/careers/*` if their effective modules include any careers module ID. The `canAccessCareers` / `canAccessCaregiving` helpers remain valid for `CAREER_DEV_OFFICER` and `CAREGIVER` respectively but are no longer the sole access gate for new roles.

### Route protection

| Route | Allowed roles |
|---|---|
| `/super-admin/*` | SUPER_ADMIN |
| `/admin/*` | ORG_ADMIN |
| `/dashboard`, `/training/*`, `/careers/*` | Leaf roles |
| `/children/*`, `/reports/*` | CAREGIVER |
| `/change-password` | Any authenticated user with `mustChangePassword: true` (middleware intercept) |
| All others (unauthenticated) | Redirect to `/login` |

ORG_ADMIN and SUPER_ADMIN do not access `/dashboard` ‚Äî they have their own portals. The `/dashboard` route returns 403 / redirects to the appropriate portal for those roles.

## 3. SSO Sign-In Policy

The existing `signIn` callback in `lib/auth.ts` currently creates new SSO users on first sign-in (upsert behaviour). With multi-tenancy, a new SSO sign-in cannot be assigned to an org automatically ‚Äî **new SSO users must be pre-created by an Org Admin or SUPER_ADMIN**.

**Updated `signIn` callback behaviour:**

- If the user **already exists** in the DB (identified by email): allow sign-in, check `user.organisation.active` (if org exists), proceed.
- If the user **does not exist** in the DB: block sign-in and return an error ‚Äî `"Account not found. Please contact your organisation administrator."` This prevents SSO users from self-registering outside of the org onboarding flow.

This means the Google and Azure AD providers will no longer auto-provision new accounts. Existing SSO users (created before this migration) are unaffected ‚Äî they will be assigned to the Legacy org in the migration step.

## 4. Session / JWT Changes

The `mustChangePassword` field must be surfaced onto the JWT and session so middleware can read it without a DB call on every request.

**In `lib/auth.ts` JWT callback:**

The `user` object is only populated on initial sign-in, not on subsequent JWT refresh cycles. The assignment must be conditional:

```ts
// Only set on initial sign-in; on refresh, preserve the existing token value
if (user) {
  token.mustChangePassword = (user as any).mustChangePassword ?? false
}
// token.mustChangePassword carries forward unchanged on refresh cycles
```

**Prerequisite:** `next-auth >= 4.22.0` is required for `unstable_update`. Verify the installed version before implementing the `/change-password` page. If the version is below 4.22, the fallback is to call `signOut()` after a successful password change and redirect to `/login` with a `?passwordChanged=1` query param ‚Äî the user signs back in with their new password and the fresh JWT will have `mustChangePassword: false`.

**In session callback:**

```ts
session.user.mustChangePassword = token.mustChangePassword
```

**TypeScript:** Extend `next-auth` types to include `mustChangePassword: boolean` on `Session['user']`.

**Middleware intercept:** In the existing NextAuth middleware (or `app/(dashboard)/layout.tsx`), after authentication check, redirect any user with `session.user.mustChangePassword === true` to `/change-password` regardless of the route they requested.

## 5. Super Admin Portal (`/super-admin`)

### Overview page

- Total orgs, total users, total lessons completed across all orgs
- Summary table: org name | user count | overall completion %

### Organisations tab (`/super-admin/organisations`)

- List all orgs with name, slug, active toggle, user count, created date
- **Create org form:** name, slug (auto-generated, editable), allowed roles (multi-select from the five leaf roles only ‚Äî `CAREGIVER`, `CAREER_DEV_OFFICER`, `STUDENT`, `INTERN`, `EMPLOYEE`; `ORG_ADMIN` and `SUPER_ADMIN` are explicitly excluded from this list and from API validation), allowed modules (multi-select from all 9 modules), active toggle
- **Edit org:** same fields
- **Deactivate org:** sets `active: false` ‚Äî enforced in the NextAuth `signIn` callback by checking `user.organisation.active` via a DB join. Users already signed in will retain their session until it expires; they will be blocked on the next sign-in attempt.
- **Delete org:** blocked via API validation if the org has any users ‚Äî returns 409 Conflict. The SUPER_ADMIN must reassign or delete all users first. This prevents silent data orphaning. (Hard delete is only permitted on empty orgs.)
- Click into an org ‚Üí scoped user list for that org

### Create Org Admin (`/super-admin/organisations/[orgId]`)

1. SUPER_ADMIN clicks "Add Org Admin"
2. Form: name, email, temporary password (plaintext input ‚Äî bcrypt-hashed on save)
3. On save: `mustChangePassword: true`, `role: ORG_ADMIN`, `organisationId` set
4. On first login: middleware intercepts and redirects to `/change-password`

### Global Announcements tab (`/super-admin/announcements`)

- List with active status, expiry, created date
- **Create form:** title, body (markdown), optional expiry date, active toggle
- Edit, toggle active, delete

### Reports tab (`/super-admin/reports`)

- Cross-org table: org √ó module ‚Äî completions / total users / %
- Filter by org, date range
- Sourced from `TrainingProgress` joined via `user.organisationId`

## 6. Org Admin Portal (`/admin`)

Replaces the existing admin page. All data scoped to `session.user.organisationId`.

### Users tab (`/admin/users`)

- Table: name, email, role, active, modules assigned, lessons completed
- **Create user:** name, email, role (limited to `org.allowedRoles`), temporary password (`mustChangePassword: true`), module checkboxes (org's `allowedModuleIds` pre-ticked, individually overridable)
- **Edit user:** role, active, module overrides; "Reset to org defaults" clears `user.allowedModuleIds`
- Cannot modify or delete self

### Announcements tab (`/admin/announcements`)

- Create/edit/delete org-scoped announcements
- Fields: title, body (markdown), optional expiry, active toggle

### Reports tab (`/admin/reports`)

- Per-module: module name | completions | total users | %
- Click module ‚Üí per-user completion detail (name, completed date, score)
- Scoped to own org only

## 7. First-Time Password Change (`/change-password`)

- Standalone page ‚Äî no sidebar or dashboard chrome
- Triggered by middleware redirect when `session.user.mustChangePassword === true`
- Fields: new password, confirm password (min 8 chars)
- On success: sets `mustChangePassword: false` in DB, then calls `unstable_update` (NextAuth v4.22+) to refresh the JWT token in place ‚Äî this updates `session.user.mustChangePassword` to `false` so the middleware intercept stops firing without requiring a sign-out/sign-in cycle. Redirects to appropriate home (SUPER_ADMIN ‚Üí `/super-admin`, ORG_ADMIN ‚Üí `/admin`, leaf roles ‚Üí `/dashboard`)
- Cannot be skipped

---

## 8. Dashboard Announcements (Leaf Role `/dashboard`)

Rendered at the top of `/dashboard`, above the stats row. Only visible to leaf roles (SUPER_ADMIN and ORG_ADMIN manage announcements from their own portals).

### Display rules

- Global announcements (`organisationId: null`) shown to all active leaf-role users
- Org announcements shown only to users matching `organisationId`
- Both types shown if present ‚Äî global first, then org-scoped
- Expired (`expiresAt < now`) and inactive announcements hidden
- Maximum 3 shown, most recent first
- If no active announcements: section does not render

### UX

- Global announcements: blue banner cards
- Org-scoped announcements: amber banner cards
- Each card: title, rendered markdown body, posted date
- Dismissible per-session via `localStorage` keyed by announcement ID (reappears on next login)

## 10. Implementation Phases

### Phase 1 ‚Äî Foundation
- Raw SQL migration (ADMIN ‚Üí SUPER_ADMIN), create Legacy org, assign existing users
- Schema changes + `prisma db push`
- `lib/rbac.ts` updates (new helpers, update existing)
- JWT/session callback extended with `mustChangePassword`
- Middleware: force `/change-password` if `mustChangePassword`
- Middleware: post-login redirect by role
- Route protection for `/super-admin` and `/admin`

### Phase 2 ‚Äî Super Admin Portal
- `/super-admin` layout + overview page
- Organisation CRUD
- Create Org Admin with first-time password (`/api/super-admin/organisations/[orgId]/admins`)
- Global announcements CRUD

### Phase 3 ‚Äî Org Admin Portal
- `/admin` rewrite: user management + module assignment
- Org announcements CRUD
- Reports tab (org-scoped)

### Phase 4 ‚Äî User Experience
- Dashboard announcements component (`/api/announcements/active`)
- `getEffectiveModules` wired into training/careers pages
- `/change-password` page
- Cross-org reports for SUPER_ADMIN

---

## 11. Out of Scope

- Email notifications when an org admin is created (password communicated manually)
- SSO/OAuth login changes for org users (existing Google/Azure SSO remains available)
- Billing or subscription management per org
- Org logo uploads (field reserved, no upload UI)
- Rich text editor for announcement body (markdown string input)
- Changes to `PasswordResetToken` model ‚Äî it remains unchanged and is not org-scoped; password reset by email continues to work for all users regardless of org
